软件工程管理

Posted on 2016-10-19

软件工程管理 容国平 CMM/CMMI

CCMI不是敏捷的对立面。所有的商业项目都是有计划的。不能认为敏捷没有计划。

CMM与CMMI

CMM1993年提出，现已弃用。

一种模式

一种过程改进的路线图，一个评估模式，不是开发模式。是帮助改进开发的。

不是一个标准

两种表现形式

阶段式

教条式的 maturity 有无

连续式

单个过程域
更灵活 capability 好坏

capability

maturity

做没做：maturity

做的结果：capability

三种不同的应用领域

系统工程（偏硬件，比如电器设备），人力资源管理。所有结构化描述的任务，即可以定义一个过程的工程，分成三种：

CMMI-DEV(CMMI for development)

CMMI-SVC(CMMI for Service)

  • SAAS: 一种软件发布和部署的方式。
  • XAAS: 比如PAAS，平台级服务。IAAS：基础设施及服务。

CMMI-SVC和CMMI-DEV在十六个过程域是相同的，但是它会更关注容灾和灾难恢复。

制约电子商务发展的重要因素是 信任度。

早期的电子商务产品是书。。

CRM：客户关系管理系统。

开源软件的质量，包括银行的很多底层软件都是开源的。这种开源更多的是一种服务。

CMMI-ACQ(采购)

系统集成：采购软硬件，组合，配置，形成一套解决方案。比如智能楼宇，智能家庭，给单位架设IT基础设施。这种适应于：重点不是开发而是选择正确的组件并正确的组合。

每一个项目不一定仅仅只包括上面的某一种。

同一个项目的客户有可能不同需求：

  • 客户希望系统在客户服务器上安装
  • 客户希望在公用服务器上安装
  • 客户希望合作搭建一个发布平台

三个模型共享十六个核心过程域

过程域是对最佳实践进行分组的结果。分成一组的实践相互间有紧密联系，是为了共同实现一个目的。

核心域在CMMI技术报告里。

四个类别

CMMI-DEV 大致分成两个类别：

  • 计划、跟踪：面向团队

    比如：配置管理。

  • 设计、编码、测试：面向这个项目

  • 改进

    比如工具引用

Humphry分成了四大类别：

项目管理类型 project management

估算、计划、跟踪

关注单个项目的实现

过程管理类型 process management

所有项目的过程方法（过程模型）

工程实践类型 engineering

支持类型 support

配置管理、度量和分析

一个过程域一定唯一的属于某个类别

四个级别

阶段式，maturity

一级：没有任何过程域

  • CMM：initial 初始级
  • CMMI：initial

没有规划，没有跟踪。没有任何评估的，开发比较混乱，没有计划；返工或救火盛行；项目成功或失败依赖个别程序员。或者计划不现实和粗略

！ 连续式的表现形式中单个过程域的第一级是performed（过程域中期望做的实践都做完了） 第零级： incomplete

二级

  • CMM中：Repitive 可重复级
  • CMMI：Managed 已管理级

什么叫管理：三个要素：有明确清晰的目标定义、有状态跟踪、纠正偏差的措施。即目标、状态和纠偏。

有规划。在项目小组这一层有很明显的项目管理特征。做估算，计划和监控。并且可以反复使用。（可重复）

但救火文化依然盛行。

一级和二级代表了一种项目管理的方法：Reactive，反应式的，出了问题之后解决问题。

敏捷方法基本是二级。技术负债，只针对目前。但是不能绝对，因为重构是着眼未来的，考虑到未来可能会做的扩展。

三级

  • CMM中： defined 已定义级
  • CMMI中：defined 已定义级

Preactive

什么是defined：组织级有一套标准流程，每一个项目组根据当前特征在组织级的流程上定义(defined)、裁剪。

  • OSSP：组织级软件过程。
  • PDSP：项目组定义的软件过程。Program Defined software process.

发展动机：促成共享。不管是好的经验还是坏的经验。定义一个标准流程。EPG小组审核好的实践，提升为组织级标准流程的一部分。让分享变成必然。

四级

  • CMM中：Managed 已管理级
  • CMMI中：quantitively managed 已定量管理的

要拿出数据出来。

进度偏差20%要偏差之类的其实是二级的特征。

为什么进度落后要加班？

如果不加班，项目最终会落后。根据预测模型（输入是当前状态，输出是项目最终的状态）。

凭什么确定预测模型是正确的？

根据大量历史数据。

结论：

这个预测模型是个高等级的模型，CMM可以让模型更可靠。

如何让模型更可靠？

  • 增加数据量
  • 分类。不同项目用不同数据
  • 消除异常数据

预测能力增强。标准差缩小。

更关注目标的达成。（像开飞机）

五级

  • CMM中：Optimizing 持续优化级
  • CMMI: Optimizing

消除影响过程的一般因素。

处理期望值。让期望值在新的水平上稳定。

找到问题的根本原因，消除这个根本原因，避免再次发生，从而提升生产水平。

总结

二级往下，只关注眼前。

二级到三级：定性到定量。

四级以上，关注结果。

敏捷：适应，ASD Adaptive Software Development

使用敏捷开发方法的企业具备的成熟度水平处于二级。

O开头的是过程管理的，反之成立 Organizational

！ CMMI中的每个过程域在阶段性的表现形式下有五个成熟度级别

！ 在连续性的表现形式下有六个能力级别（0级：incomplete 1级：performed 2级：managed 3级：defined 4级：quantitive managed 5级：optimizing）

六个连续式表现形式下的能力水平

在连续性的表现形式下有六个能力级别（0级：incomplete 1级：performed 2级：managed 3级：defined 4级：quantitive managed 5级：optimizing）

0级

某个过程域中有一些实践做了，有一些没有做。做得不够完整

1级

工作都做了 performed

至此模型会严重不一致

连续式和阶段式仅仅表现得是两种不同的过程改进路线图，但是最终的结果应该是一致的，应有一个Mapping规则：

  • 成熟度水平为二级那么所有的二级过程域（七个）的能力水平为二级。
  • 三级则是所有的二级和三级过程域（十八个）的能力水平达到三级。
  • 四级则是所有的二级和三级和四级过程域（二十个）的能力水平达到三级。
  • 五级则是所有的过程域（二十二个）的能力水平达到三级。（原有版本：“部分过程域达到能力水平四级或五级”，现已去掉）

为什么？

  • 能力模型的四级与五级的要求与成熟度四级与五级的过程域要求是几乎一致的。只要到三级，就可以达到四级与五级的水平。

过程域

一组相关的实践的集合。

  • Specific Goals：必须。特殊目标。只可能在本过程域中出现
  • Genetic Goals：必须。在所有的过程域都出现，只是表述稍微有差异
  • Specific Practice：最佳实践。希望实现的。为了支持某特殊目标而采取的特殊实践。只要不影响目标达成，可以使用替代实践。不一定要用最佳实践。
  • Genetic Practice（GP）：最佳实践。希望实现的。为了支持所有的通用目标而采取的实践。
  • Typical Work Products: 补充信息类。典型工作产品。
  • Subpractice：补充信息类。推荐做法。

文档属于补充信息的内容。

高等级的要求做到Informative层。

评估方法

  • 发现式：评估小组找证据。
  • 验证式：公司提供材料（文档），由评估小组检查。

验证式更省事，所以大部分评估都采用文档验收。

所有过程域的GP

  • GP2.1：建立指导原则。
  • GP2.2：为某一个过程域的开展进行策划。建立规划。
  • GP2.3：资源
  • GP2.4：清晰的角色分配
  • GP2.5：培训
  • GP2.6：管理变更
  • GP2.7：干系人如何协作
  • GP2.8：按照规划检查 监控
  • GP2.9：过程域开展实施状况的客观检查（第三方检查）
  • GP2.10：高级管理者参与
  • GP3.1：组织级的规范
  • GP3.2：结果保存，变成组织级的资产。支持后续过程改进

具体过程域

PP project planning 项目策划

项目管理、二级

  • SG1: Establish Estimate
    估算

    估算是为了得出计划

    和方法无关，但要有。可以用重量级的也可以用轻量级的开发方法，但是一定要有一个估算。

    估算内容：

    • 定义（估算）项目范围 scope Typical Work Product 会产生一个工作分解结构(WPS)
    • 定义项目生命周期 可能会用几个迭代，如何定义每个阶段 决定了计划的结构 决定了项目计划的样子 不同迭代的计划会不一样
    • 工作产品和任务的属性

    • 需要付出的工时是多少

      COCOMO模型–估算方法

      估算方程： y = ax^c + d

      其中，y是时间（人天/人月），X是规模（代码行，功能点）a\c\d是系数

      Scrum的估算结果是一个Scrum Point 一个scrum point具体是什么已经被虚拟化了，只是一个相对大小的比较。是指不同的两个工作量之间的差值。比如一个小房间需要3个point的油漆，比它大的需要4个。定义某一项工作有无数种选择。用费波纳契亚数列来估算，里面每一个都是可以的，都是正确的。

      PSP里面支持五种：非常小，小，中等，大，非常大。

      可以看出，估算都是很模糊的，不精确的，不能估算地太准确。由软件开发的性质决定。在传统行业中可以把工作理解为几乎完全一样的动作（过程）的重复，当动作稳定后，波动是非常小的。但是软件不是这样的。大模块不是小模块的线性叠加。因为里面有人类的思维过程。

      如何估算？？只要估算出的数字大家都会接受，就可以了。

      估算结果的获得：

    • 上面的公式 y = ax^c + d

    • 直接猜测 和公式没有统计上的差别 乔布斯 现实扭曲力场

      冒泡排序：一个函数和两个函数的估算哪个会更准确？=>倾向于两个方法更准确，因此按照两个函数来估算，因为大家都这么以为。

      估算不是追求一个客观真实的结果，而是为了一个大家都相信的数字

  • SG2: Develop a Project Plan 制定计划

    • 预算和日程计划
    • 项目风险

    • 项目数据管理计划

      配置管理是其中的一部分

      项目的熵在不断增加，会越来越混乱，需要配置管理

    • 项目资源规划 （人力资源，有效时间）

      有效工作时间会比直观的感觉低很多。有效时间：做的工作可以直接对应到日程表上的任务。处理变更不算有效时间，因为它不在日程表上。修改错误的时间也不算有效时间。

    • 培训计划 相关干系人的参与计划（客户和高级管理者）

      在项目里程碑的时候邀请干系人参与

  • SG3: Obtain Commitment to the Plan 评审

    • 对计划和相关参与人进行Review，使大家都认可。为了追求一种信任感
    • 协调内容和资源水平
    • 获得一致认可

PP的GP2.2是什么？有什么现实意义？

为计划做计划。需要协调人员时间和资源。

现实中，项目计划的计划不是临时定义的计划，是固定好的一个模板。规定了会议等的时间和流程，以提前协调人员。

因此是有现实意义的。

PMC Project Monitoring and Control 项目管理与监控 二级

状态跟踪与纠正

识别偏差->纠正

如何识别偏差：与计划相比较

  • SG1：MOnitor Project Against Plan

    与计划的偏差

    • Monitor Project Planning Parameters 项目策划的参数

      范围、资源水平、估算规模、

    • Monitor Commitments

      尤其是程序员的承诺，程序员往往过于乐观。

    • Monitor Project Risks

      见风险管理

    • Monitor Data Managemeng

      有效的降低熵的手段 文件结构清晰、命名清晰

    • Conduct Milestones Reviews

      里程碑评审

      这个项目能否进展到下一步

    • Conduct Progress Reviews

      进度评审

      判断项目是否在按照计划进展，要关心质量、日程、风险等等的进度

      如何分清上面两个的区别？

      • 实际进展和计划相比较是否正常=>项目进度 范围小，频率高 小组例会即可，不需要包括客户等等人员，会议里需要回答上一周期完成了什么，下一周期要实现什么
      • 这个项目需不需要进行下去=>里程碑 需要客户和高级经理参与 范围大，频率小 比如需求结束以后，发现出现了很多错误。因此需求这个里程碑完成但是质量不好，不可以进入下一个阶段，需要推倒重来。不可以项目经理一个人定的，需要高级经理和客户决定。关系到项目是否延期和是否直接结束。
      • 它们关心的内容是很接近的。

    • Monitor Stakeholder Involvement

      相关干系人的参与

  • SG2: Manage Corrective Action to Closure

    纠正偏差

    • Analyze Issues

      分析问题的根源。弄明白根本原因，对症下药。

      不同的场景下的原因是不同的。

      比如一个项目的进度落后了，原因有哪些？

      • 需求变更
      • 估算过于乐观（加班可能没有用了；或者人力投入不足）

    • Take Corrective Action

    • Manage Corrective Action 管理纠偏的措施

向一个已经进度落后的项目里增加人员只会导致项目进一步落后 ——《人月神话》

PMC的GP2.8是什么？有什么现实意义？

监控的监控。

  • 原本定好的活动不再做了，需要监控。

现实中，很可能虽然有周例会这种制度，但是已经不再开了。需要有一个制度来监控是否进行了周例会。

  • 针对已发现的问题的分析，统计和管理

什么时候开展PMC的GP2.8？由谁来做？

  • 里程碑的时候进行GP2.8. 可以提出周例会等问题。
